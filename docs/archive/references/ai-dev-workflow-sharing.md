# AI 辅助开发 Web 应用：从 0 到 1 的完整流程

> Historical sharing material. This file preserves an earlier explanation of the AI development process and is not an active routing contract.
>
> Current workflow routing lives in [Pantheon Workflow Routing](./WORKFLOW_ROUTING.md). Mentions of `superpowers`, `gstack`, or older OpenSpec-first chains below are presentation context, not current execution instructions. `impeccable` remains the active UI visual quality gate.

## 1. 这次分享想讲什么

这份分享只讲一条线，怎么把 AI 真正放进项目里，而不是停留在聊天层。

`prompt -> context -> harness -> 交付`

核心意思很简单：

- `prompt` 负责说明目标
- `context` 负责说明规则和背景
- `harness` 负责把 AI 的输出纳入工程流程，保证能验证、能回滚、能协作

还有一个被实践反复验证的结论：

- 真正决定 AI 协作质量的，不是多说几句“继续”或“按最佳实践做”
- 而是有没有把项目边界、默认规则、停点和验收方式提前沉淀成项目文档

在这个链路里，`skills` 和 `MCP` 是两种不同的扩展方式。

如果只讲 prompt，大家很容易以为 AI 开发就是“多试几次”。
实际不是，真正决定结果的是上下文、约束、流程和验收。

### 先给结论

我们现在这套做法，核心不是“让 AI 写代码更快”，而是：

- 先用 OpenSpec 管需求和变更身份
- 先用文档把问题定清楚
- 再用 superpowers 把方案想清楚
- 再用 task packet 把事情拆清楚
- 如果触碰 UI，先过 impeccable 视觉质量门
- 再让 AI 按 harness 和门禁去实现
- 最后用验收把结果收口

### 一条总流程

旧版分享稿中的流程链已废弃。当前执行链路以 [Pantheon Workflow Routing](./WORKFLOW_ROUTING.md) 为准；其中 `impeccable` 仍是 UI 视觉质量门。

### 让 AI 越来越懂你的 5 件事

结合这段时间在 Pantheon 上的实践，真正有效的不是更复杂的 prompt，而是下面 5 类稳定上下文：

1. 明确仓库角色：`pantheon-base` 是底座，`pantheon-ops` 是业务仓，`harness-engineering` 是方法与工具仓。
2. 明确落点规则：共享问题先改 base，再同步 ops；业务问题留在 ops。
3. 明确任务模式：是评审、实现、UI、同步、冒烟，还是文档整理。
4. 明确必读文档：先看 contract、design、inheritance、acceptance，再看代码。
5. 明确收口口径：代码、测试、i18n、菜单、权限、文档、证据哪些要同轮完成。

这 5 类上下文一旦固定下来，AI 的输出会比单纯“多聊几轮”稳定得多。

### 一页看懂

| 词 | 作用 | 你可以把它理解成 |
|---|---|---|
| `prompt` | 说清楚这次要做什么 | 任务描述 |
| `context` | 说清楚项目规则和边界 | 项目背景包 |
| `openspec` | 给需求和变更一个正式身份 | 需求管理入口 |
| `design / contract` | 说清楚什么能做、什么不能做 | 正式依据 |
| `superpowers` | 先把方案想清楚 | 规划稿 |
| `task packet` | 把方案压成一次可执行任务 | 任务单 |
| `impeccable` | 给 UI 加视觉和状态质量门 | 视觉门禁 |
| `harness` | 让 AI 按工程流程交付 | 控制系统 |
| `skills` | 复用做事方法 | 方法包 |
| `MCP` | 接外部工具和系统 | 工具接口 |
| `gstack / 门禁` | 跑验收和留证据 | 执行和验收层 |
| `AGENTS.md` | 定义仓库级规则 | 项目约束说明 |

## 2. 什么是 vibe coding

vibe coding 可以粗略理解成：

- 不是自己从头手写每一行代码
- 而是把需求讲清楚，让 AI 帮你生成、修改、补全代码
- 你负责判断方向、边界、验收和修正

它不是“不要工程规范”，恰好相反。
越依赖 AI，越需要工程化。

## 3. 一个新人能理解的开发流程

### Step 1: 先说清楚要做什么

先定义业务目标，不要一上来就让 AI 写代码。

例如：

- 这是一个后台管理 Web 应用
- 需要登录、菜单、权限、列表、表单
- 用户是内部团队，不是普通消费者

### Step 2: 给 AI 足够 context

context 不是废话，是项目规则。

至少要告诉 AI：

- 项目是什么
- 技术栈是什么
- 目录怎么分
- 哪些能力属于底座，哪些属于业务
- 什么能改，什么不能改

### Step 3: 拆成小任务

不要一次性让 AI 做完整项目。

建议按这个顺序：

1. 页面骨架
2. 接口定义
3. 数据模型
4. 核心逻辑
5. 测试和验收

### Step 4: 验证

AI 写完不等于完成。

必须看：

- 能不能跑起来
- 接口对不对
- 页面能不能用
- 权限有没有问题
- 有没有测试或截图证据

### Step 5: 收口

把这次产出变成可复用资产，而不是一次性聊天记录。

包括：

- prompt 模板
- context 说明
- task packet
- 测试命令
- 验证结果

这样下次做同类需求时，不用重新从零解释一遍。

## 4. Prompt 怎么写

一个好 prompt 至少包含 5 件事：

1. 目标
2. 约束
3. 输入
4. 输出格式
5. 验收标准

### 简单模板

```text
请基于当前项目实现一个 XX 页面。
要求：
1. 使用现有技术栈
2. 保持现有目录结构
3. 不要改动无关模块
4. 先给出方案，再给出实现
5. 完成后说明如何验证
```

### 不好的 prompt

```text
帮我写个页面
```

问题是 AI 不知道：

- 页面属于什么业务
- 依赖哪些接口
- 风格和约束是什么
- 做到什么程度算完成

## 5. Context 应该放什么

可以把 context 理解成“项目说明书的精简版”。

建议包含：

- 项目定位
- 技术栈
- 目录结构
- 领域边界
- 设计规范
- API 约定
- 权限模型
- 测试方式

### 一个更实用的 context 包

如果是内部团队协作，建议固定成 4 块：

1. 项目定位
2. 目录结构
3. 业务边界
4. 验证方式

例如：

- 这是 base 还是 ops
- 这是平台层、系统域，还是业务域
- 能不能改底座
- 改完怎么测

再往前走一步，建议直接固定成一个“最小任务包”，以后重复使用：

```text
目标仓库：
层级：
任务模式：
先读：
实现范围：
同步要求：
验证方式：
停点：
```

这样做的好处是，AI 不需要再反复猜：

- 这次是改 base 还是改 ops
- 是先做设计还是直接实现
- 改完要不要同步另一仓
- 是本地验证就行，还是要跑 smoke / 截图 / 验收矩阵

### 例子

如果你要做后台系统，context 里要明确：

- 登录、权限、菜单属于底座
- 业务模块不要直接耦合底座内部实现
- 前端页面要复用统一组件
- 文案、错误、菜单标题要统一处理

## 6. 文档怎么管理，为什么先讲文档

在我们现在这套体系里，文档不是补充说明，而是开发入口。

原因很简单：

- 没有文档锚点，AI 不知道该信什么
- 没有边界文档，AI 很容易改错层
- 没有验收文档，代码改完也很难判断是否真正完成

### 我们现在的文档治理主线

不是所有文档都平铺，而是按一条主链来走：

`Contract -> Design -> Assessment -> Remediation -> Acceptance`

这条链解决的是：

- 需求挂在哪里
- 设计以什么为准
- 偏差怎么评估
- 问题怎么整改
- 最后怎么验收

### 为什么这样管理

因为很多团队的常见问题是：

- 设计稿很多，但没有主依据
- 代码已经改了，验收口径还没定
- 老文档和新文档并存，没人知道该看哪份

文档治理本质上是在解决“信息漂移”。

### 讲给团队最简单的说法

对 AI 来说，文档不是附件，而是正式上下文来源。  
AI 不应该只看一次聊天描述，而应该优先看：

- contract
- design
- acceptance
- AGENTS.md

这也是为什么你现在这套方法里，文档管理和开发流程是连在一起的。

## 7. 什么是 harness

harness 就是“让 AI 变成可控工程流程”的那一层。

它通常包括：

- task packet：任务单
- evidence：证据
- review loop：审查闭环
- checkpoint：断点恢复

它解决的问题不是“AI 会不会写代码”，而是：

- 这次改动范围是什么
- 改完怎么验证
- 出问题怎么回滚
- 别的同学怎么接着做

### 一个最小可用的 harness

对技术团队来说，不需要一开始就做很重的体系。
先有这 3 个东西就够用：

- 任务说明
- 验证命令
- 结果记录

这三样齐了，AI 产出才算进入工程流程。

### Harness Engineering 的核心思想

可以用一句话概括：

`coding agent = model + runtime + harness`

意思是：

- `model` 是大模型本身
- `runtime` 是 Claude Code、Codex 这类执行环境
- `harness` 是项目自己定义的控制系统

真正稳定的 AI 开发，不是只换更强模型，而是把 harness 建起来。

### 为什么要讲 Harness Engineering

因为团队一旦进入真实项目，就会遇到几个典型问题：

- prompt 每个人写法不一样，结果不稳定
- 同一个需求，换个人问 AI，改动位置可能完全不同
- AI 写完代码，但没有验证证据
- 一轮对话结束后，后续成员接不上

Harness Engineering 的目标，就是解决这些问题。

### Harness Engineering 想解决什么

它本质上是在做 4 件事：

1. 定义任务边界
2. 约束 AI 修改范围
3. 固定验证方式
4. 保留可追踪证据

所以它关心的不是“模型聪不聪明”，而是：

- 任务怎么进入系统
- AI 改了什么
- 改完怎么证明有效
- 下一轮怎么继续

### 在我们项目里，harness 主要体现在哪

在 Pantheon 这套仓库里，harness 不是抽象概念，而是具体文件和流程：

- `docs/harness/*`：定义通用协议
- `TASK_PACKET_SPEC`：定义非 trivial 任务怎么写任务单
- `VERIFICATION_EVIDENCE_SPEC`：定义验证证据怎么记录
- `REVIEW_LOOP_SPEC`：定义怎么 review 和收口
- `AGENTS.md`：定义仓库级行为约束

也就是说：

- prompt 只是入口
- harness 才是长期可复用的方法

### 一个更容易讲给团队听的版本

如果只靠 prompt，AI 开发更像“临场发挥”。

如果有 harness，AI 开发才更像“按工程流程交付”。

这也是为什么我们现在强调：

- task packet
- evidence
- review
- checkpoint

这些东西不是文档负担，而是让 AI 开发可协作、可审计、可回放。

## 8. Superpowers 是什么，怎么接进流程

superpowers 可以理解成“更高一层的设计和规划入口”。

它不负责直接写代码，主要负责把问题先想清楚：

- 先定义目标
- 先判断边界
- 先把方案写成可讨论的设计稿
- 先把后续实现的约束写明白

### 它在流程里的位置

更准确地说，superpowers 站在这条链路的前半段：

`需求 -> 设计文档 / plan -> task packet -> 实现 -> 验证`

它的作用是把“想法”变成“可以交给 AI 或人执行的计划”。

### 为什么要有 superpowers

因为复杂需求如果直接进代码，容易出这些问题：

- 边界没定清
- 设计没收口
- 代码改到一半才发现方向错了
- 后面的人不知道为什么这么做

superpowers 的价值，就是先把这层前置掉。

### 在我们的文档体系里

你现在这套文档已经把这件事串起来了：

- 设计文档负责定义方向
- contract 负责定义边界和验收口径
- superpowers plan 负责把设计转成可执行计划
- task packet 负责把计划压成一次具体任务
- evidence / review 负责把结果闭环

也就是说，superpowers 不是额外负担，而是把“从想法到执行”的中间层固定下来。

## 9. OpenSpec 是什么，为什么适合做需求管理

如果说 contract 和 design 是正式依据，那么 OpenSpec 更像“变更管理入口”。

它最适合放在流程最前面，做这几件事：

- 给需求一个正式 change identity
- 把 proposal、tasks、spec delta 挂到同一条链上
- 让后面的 superpowers、task packet、evidence 都有可追踪来源

### 为什么需求管理优先用 OpenSpec

因为真实项目里最容易乱的，不是代码，而是需求版本。

常见问题是：

- 大家都知道要改什么，但没有统一 change identity
- 方案改过几轮，没人知道哪版还有效
- task packet 已经开工了，但需求边界还在聊天里漂
- 后面 review 和 acceptance 找不到最初依据

OpenSpec 的价值，就是把“这次到底在改什么”先钉住。

### 它在流程里的位置

更稳的顺序是：

`需求想法 -> OpenSpec change -> contract/design -> superpowers plan -> task packet`

也就是说：

- OpenSpec 不替代 design
- OpenSpec 也不替代 task packet
- 它负责先把这次变更立项、命名、挂链

### 讲给团队最简单的一句话

如果 superpowers 负责“把方案想清楚”，  
那 OpenSpec 负责“先把这次需求挂对地方”。

### 在方法上怎么用

对 non-trivial 任务，建议默认这样处理：

1. 先判断这是不是一个值得追踪的 change
2. 在 `openspec/changes/<name>/` 下挂这次变更身份
3. 再把 contract、design、proposal、tasks 串起来
4. 后续 task packet 明确引用对应的 OpenSpec change
5. evidence 和 review 继续沿用同一个 change linkage

这样后面看任何一个任务，都能回答 3 个问题：

- 它为什么开始
- 它按什么设计实现
- 它最后怎么被验证和收口

## 10. Skills 和 MCP 是什么

### Skills

skills 可以理解成“可复用的做事方法包”。

它通常会包含：

- 一段固定流程说明
- 项目规则
- 参考文档
- 脚本或模板

在 AI 开发里，skills 的作用是把常见工作固化下来，比如：

- 生成页面
- 写测试
- 做 review
- 跑验证

它解决的是“这类事应该怎么做”。

### MCP

MCP 可以理解成“让 AI 接外部工具和外部系统的接口层”。

它让 AI 不只会看文本，还可以接：

- 文件系统
- 浏览器
- 数据源
- 内部服务
- 其他工具

它解决的是“AI 还能连什么工具来做事”。

### 两者区别

- skills 偏方法和流程
- MCP 偏工具和连接

简单说：

- skills 让 AI 做事更像“按规矩办事”
- MCP 让 AI 能碰到更多真实系统

## 11. impeccable 是什么，为什么它应该做 UI 视觉门禁

如果 OpenSpec 管的是需求身份，  
那么 impeccable 管的是 UI 质量底线。

它不是“把页面做漂亮一点”的建议包，而是视觉质量门禁。

### 它主要管什么

只要任务碰到这些内容，就应该触发 impeccable：

- 页面布局
- dashboard/admin/workbench
- 表格、表单、图表
- 导航、弹窗、抽屉、工具栏
- loading / empty / error / forbidden 等状态
- desktop / mobile 响应式表现
- 颜色、字体、间距、图标、动效

### 为什么要单独把 impeccable 拿出来

因为前端常见的问题不只是“功能没做完”，还有很多“看起来差不多”的假完成：

- 文本溢出或遮挡
- hover / loading 导致按钮跳动
- 主按钮不明显
- empty / error 状态缺失
- mobile 下横向滚动
- 焦点态和可访问性缺失

这些问题只看代码很难发现。

### 它在流程里的正确位置

对 UI 任务，比较稳的顺序不是“代码写完再顺便看看”，而是：

1. 先判断 UI surface 是什么类型
2. 先定义目标视觉感受
3. 再实现页面
4. 用 impeccable checklist 做 review
5. 用 gstack 或截图拿 rendered evidence
6. 最后再决定是否算完成

也就是说，impeccable 不只是 review 工具，它还是完成定义的一部分。

### 它和 gstack 的关系

- `impeccable` 负责定义“应该看什么”
- `gstack` 负责提供“真实页面证据”

一个管质量标准，一个管运行态留证。

### 讲给团队最简单的一句话

OpenSpec 解决“需求有没有挂清楚”，  
impeccable 解决“UI 有没有真的过关”。

## 12. gstack 和前端门禁怎么接进来

### gstack 是什么

gstack 可以理解成“执行和验收的运行层”。

它擅长做这些事：

- 打开页面
- 注入登录态
- 跑浏览链路
- 看 console error
- 截图
- 采集验收证据

它解决的是“代码看起来对不对”之外的那一层，真正跑起来后对不对。

### 为什么前端必须有门禁

因为前端最容易出现一种假完成：

- 代码提交了
- 页面也能打开
- 但文本硬编码了
- 菜单不对
- i18n 没接
- 状态不全
- 视觉和交互不一致

所以前端不能只靠人工印象，要有门禁。

### 你这套门禁本质上在管什么

本质上是在管这几类事：

1. 文案规范
2. 菜单和路由契约
3. i18n 约束
4. 页面状态完整性
5. 构建和 smoke 验证
6. 浏览器证据

### 它怎么嵌进 AI 开发流程

一个比较稳的顺序是：

1. 先有设计文档和 contract
2. 再有 superpowers plan
3. 再拆 task packet
4. AI 实现代码
5. 用 gstack 或 smoke 跑页面
6. 触发前端门禁检查
7. 留下证据和验收记录

这样 AI 不只是“写了前端”，而是“按前端规则交付了前端”。

### 这对团队意味着什么

意思是，AI 不能只会生成页面结构，还得过你定义的工程门槛：

- 不能随手硬编码
- 不能绕开菜单和权限
- 不能省略状态和错误处理
- 不能跳过浏览器证据

这就是门禁的价值。

## 13. `AGENTS.md` 约束到底在管什么

这里说的 `agent.md`，在我们仓库里实际对应的是 `AGENTS.md`。

它不是“写给 AI 看的一段备注”，而是项目级行为约束。

### 为什么需要 `AGENTS.md`

如果没有这层约束，AI 很容易出现这些问题：

- 把系统域和业务域混在一起改
- 为了快，把业务逻辑直接塞到底座
- 把认证、IAM、组织、配置当成一个大模块处理
- 改了代码，但没同步测试、文档、菜单、权限

`AGENTS.md` 的作用，就是提前把这些误改拦住。

### 在 Pantheon 里，`AGENTS.md` 最核心的作用

可以概括成 5 点：

1. 先读上下文，再动代码
2. 先判边界，再决定改哪里
3. 保持底座和业务隔离
4. 保持前后端和权限模型一致
5. 改代码时同步补测试和文档

### 它要求 AI 先做任务分类

Pantheon 不允许一上来就“泛泛地改 system 模块”。

开始任何实现前，先判断任务属于哪一层：

- `platform`
- `system/auth`
- `system/iam`
- `system/org`
- `system/config`
- `business/*`

这一步非常关键。

因为很多错误，不是代码不会写，而是改错层了。

### 它要求 AI 理解底座边界

在 `pantheon-base` 里：

- 平台层负责应用壳、聚合页、路由装配
- 系统域负责认证、IAM、组织、配置、审计等通用能力
- 业务域通过 `business/*` 扩展

所以 AI 不能：

- 把运维业务逻辑塞进 `system/*`
- 把 dashboard 聚合页塞回某个系统子域
- 把通用能力做成某个业务模块私有逻辑

### 它要求 AI 遵守几个关键工程约束

比如：

- 后端走垂直切片，不回到大一统 service/repository 模式
- DTO 不能省，不能直接把 GORM Model 返回前端
- 前端文本要走 i18n
- 菜单和权限要解耦
- 页面新增时要同时考虑 loading / empty / error / forbidden / submitting 状态

这些约束的目的都是同一个：

让 AI 产出尽量贴合项目已有结构，而不是生成一套“看起来能跑”的外来代码。

### 在 `pantheon-ops` 里还多一层继承约束

`pantheon-ops` 的 `AGENTS.md` 比 base 多了一层重点：

- `pantheon-base` 是唯一底座知识源
- 通用问题优先回 base 修
- ops 只放自己的业务模块和继承说明

这意味着：

如果 AI 发现某个能力其实是通用平台能力，就不应该直接在 ops 里打补丁，而应该考虑先回 base 处理。

### 可以讲给团队的一个简单结论

`AGENTS.md` 的价值，不是限制 AI，而是减少错误自由度。

它让 AI 明白：

- 哪些地方能改
- 哪些地方不能乱改
- 改的时候要一起补哪些配套项

这样 AI 才不会越改越乱。

## 14. Claude Code 和 Codex 怎么理解

### Claude Code

- 更像一个能在终端里执行任务的开发助手
- 擅长读仓库、改文件、跑命令、按项目规则工作
- 适合和仓库里的规则文件、技能、命令配合使用

### Codex

- 也是 coding agent
- 重点同样是把模型接到代码仓库和工程流程里
- 适合做任务执行、代码修改、检查和验证

### 对团队来说要记住的

工具不是重点。

重点是：

- 先给规则
- 再给上下文
- 然后让 AI 做小步改动
- 最后用验证闭环收口

### 适合团队的分工方式

- Claude Code：适合持续上下文下的改代码、读仓库、跑命令
- Codex：适合做明确任务的修改、检查、验证
- 人：负责定义边界、看结果、拍板

工具可以换，流程别乱。

## 15. 现在的 pantheon-base 在做什么

`pantheon-base` 是底座，不是业务应用本身。

它负责的是：

- 平台层：应用壳、路由、聚合页
- 系统域：认证、IAM、组织、配置、审计
- 低代码治理：受控生成、模块接入治理

一句话：

`pantheon-base` 负责“后台系统通用底座”

它不是一个单纯 CRUD 壳，而是一个可复用的企业后台基础平台。

### base 里已经有的能力

- 登录和会话
- 用户、角色、权限
- 菜单和路由装配
- 组织和配置
- 审计和多语言
- 受控生成和模块治理

---

## 16. 怎么在 pantheon-base 上继续做 pantheon-ops

`pantheon-ops` 是在底座上做运维业务。

它的思路不是重做一套后台，而是：

- 直接继承 base 的登录、权限、菜单、组织、配置能力
- 在业务域新增运维模块
- 先做 CMDB
- 再做 Deploy
- 后续再扩展其他运维能力

### ops 的第一批业务

- CMDB：资源台账、分组、标签
- Deploy：部署包、任务、目标、执行记录

这两个模块的共同点是：

- 都依赖底座能力
- 都是明确的业务域
- 都不应该反过来污染 base

### 你可以这样讲

- base 是地基
- ops 是在地基上盖运维应用
- 通用能力回到底座
- 运维差异留在 ops

## 17. 一个贴近我们项目的实际例子

假设我们现在要在 `pantheon-ops` 里新增一个“主机列表”页面。

一个比较合理的 AI 开发流程是：

### 第一步：先判断边界

先问清楚这几个问题：

- 这是底座能力还是运维业务能力
- 要不要改登录、权限、菜单、组织
- 数据表是放 `system_*` 还是 `biz_*`

这个例子里，答案应该是：

- 属于 `business/cmdb`
- 数据表应该是 `biz_*`
- 权限和菜单复用底座现有能力

### 第二步：给 AI 一个明确任务

例如：

```text
请在 pantheon-ops 中新增 business/cmdb/host 主机列表能力。
要求：
1. 不修改 pantheon-base 的系统域实现
2. 后端按业务模块垂直切片实现
3. 前端页面走现有模块注册方式
4. 菜单、权限、i18n、审计点需要同时考虑
5. 完成后给出验证步骤
```

### 第三步：补 context

要告诉 AI：

- 当前仓库是 `pantheon-ops`
- 底座来自 `pantheon-base`
- 业务模块只能放在 `modules/business/*`
- 前端页面走 `src/modules/business/*`
- 不能把运维业务逻辑塞回 system 域

### 第四步：让 AI 分步干活

一个更稳的拆法是：

1. 先出数据模型和接口设计
2. 再出后端模块骨架
3. 再出前端页面骨架
4. 再补菜单、权限、i18n
5. 最后补测试和验收

### 第五步：人工验收

重点不是“代码有没有生成”，而是：

- 页面是不是挂在正确模块下
- 权限是不是走现有模型
- 菜单是不是复用现有机制
- 数据表命名是不是符合规范
- 是否影响了 base 和 ops 的边界

这个例子基本就是我们现在项目最常见的 AI 协作场景。

## 18. 这个项目的正确演进方式

不要把所有东西都塞进一个“大系统”。

建议分三层理解：

### 平台层

- 应用壳
- 聚合页
- 跨域统计

### 系统域

- 登录
- 用户
- 角色
- 权限
- 组织
- 配置

### 业务域

- CMDB
- Deploy
- 未来的其他运维模块

### 一个判断标准

如果需求是“所有业务都会用到的通用能力”，优先放 base。

如果需求是“运维场景特有能力”，放 ops。

如果需求只是某个页面、某个功能组的差异，不要上升成底座改造。

## 19. 把 OpenSpec、impeccable、superpowers、gstack 组合成一套推荐实践

如果只分别使用这些 skill，很容易变成：

- 需求在一处记
- 方案在一处想
- UI 靠感觉看
- 验收再临时补

真正稳的方法，是把它们放到固定职责位上。

### 先说结论

这套方法更准确地说，不是“放之四海而皆准的唯一最佳实践”，而是：

- 对我们这类后台 Web 项目比较稳的推荐实践
- 对团队新人门槛较低的入门实践
- 对多人协作比较容易复制的方法

也就是说，它追求的不是大而全，而是：

- 能理解
- 能上手
- 能复用
- 能逐步加严

### 一套推荐分工

- `OpenSpec`：做需求管理和变更身份管理
- `contract / design`：做正式边界和约束定义
- `superpowers`：做设计展开和执行规划
- `task packet`：把计划压成一次可执行任务
- `impeccable`：做 UI 视觉和状态质量门
- `gstack`：做运行态浏览器验证和证据采集
- `harness checks`：做 task/evidence/review 的机械收口

### 一条推荐主链

旧版分享链路已废弃；当前执行路由以 [Pantheon Workflow Routing](./WORKFLOW_ROUTING.md) 为准。`impeccable` 保留为 UI 视觉质量门。

### 这套方法最适合什么场景

最适合：

- 需求不再只是个人对话，而要团队协作
- 页面不只是能用，还要过统一视觉门
- 要留下可回看、可审计、可接手的证据
- 一个需求会跨文档、代码、浏览器验收多个阶段

### 这套方法不需要每次全开

如果每个小改动都把整套链路跑满，团队很快会觉得重。

所以更合理的做法不是“所有任务一刀切”，而是分档使用。

### 最低门槛的三挡用法

#### 第一挡：小任务

适用场景：

- 文案修正
- 小样式修正
- 单点逻辑修复
- 不影响架构和边界的局部改动

最小流程：

`prompt + context + 实现 + 验证`

要求：

- 不必强制上 OpenSpec
- 不必强制写 task packet
- 但必须留下最基本验证结果

#### 第二挡：标准任务

适用场景：

- 新增一个页面
- 新增一个接口
- 调整模块行为
- 触碰权限、菜单、i18n、审计、schema、base/ops 边界

最小流程：

当前最小流程按 [Pantheon Workflow Routing](./WORKFLOW_ROUTING.md) 选择，不再在本分享稿中定义。

要求：

- 有 change identity
- 有明确边界
- 有验证计划
- 有 evidence

#### 第三挡：UI 或高风险任务

适用场景：

- 触碰用户可见页面
- 涉及表格、表单、工作台、dashboard
- 涉及核心权限、关键流程、跨模块改动

最小流程：

当前 UI / 高风险流程按 [Pantheon Workflow Routing](./WORKFLOW_ROUTING.md) 选择；UI 任务仍必须经过 `impeccable`。

要求：

- UI 必须过 impeccable
- 必须有浏览器证据或明确说明缺失原因
- 必须记录剩余风险

### 真正落地前，只需要先准备 4 个东西

如果想把门槛压低，不要一开始就补全所有协议。

先准备这 4 个最小件就够了：

1. 一个 OpenSpec change 骨架
2. 一个 task packet 模板
3. 一个 gstack 验证入口
4. 一条“UI 任务默认走 impeccable”的团队约定

也就是说，先把入口、边界、验证、视觉门这 4 件事立起来，  
这套方法就已经能跑。

后面再逐步补：

- review 模板
- evidence 目录规范
- acceptance checklist
- 机械检查脚本

### 每一步分别解决什么问题

1. `OpenSpec`
   解决“这次需求到底是什么、挂在哪里”
2. `superpowers`
   解决“这次实现思路、边界和步骤是什么”
3. `task packet`
   解决“这次到底允许改哪些文件、怎么验证”
4. `impeccable`
   解决“UI 是否真的达到视觉和状态质量要求”
5. `gstack`
   解决“有没有真实运行证据，而不是只看代码”

### 一个更实用的落地原则

可以把它记成 5 句话：

1. 用 OpenSpec 管需求，不用聊天记录管需求
2. 用 superpowers 先想清楚，不要直接把复杂需求扔进代码
3. 用 task packet 控边界，不要让 AI 自由发挥改仓库
4. 用 impeccable 守 UI，不要把“差不多能看”当完成
5. 用 gstack 留证据，不要把“本地应该没问题”当验收

### 对团队最大的价值

这套方法最大的价值，不是让某个模型更聪明，  
而是把不同 skill 擅长的能力放在最合适的位置：

- OpenSpec 擅长挂需求主线
- superpowers 擅长做设计和计划展开
- impeccable 擅长卡住 UI 质量下限
- gstack 擅长做真实页面和浏览器证据

这样最后得到的不是“AI 写了点代码”，而是一套能持续复用的开发方法。

## 20. 我们内部落地时建议统一的最小流程

为了让组内成员都能复用，建议统一成下面这个最小流程：

1. 先判断是 `base` 还是 `ops`
2. 再判断是平台层、系统域还是业务域
3. 先判断这是小任务、标准任务，还是 UI / 高风险任务
4. non-trivial 任务先挂 `OpenSpec change`
5. 再补 contract / design / context
6. 再用 superpowers 形成 plan
7. 再写 task packet
8. 再让 AI 小步实现
9. UI 任务走 impeccable
10. 再用 gstack 或 smoke 跑验证
11. 最后记录 evidence、review 和 acceptance

如果这个顺序反过来，通常就容易出现几类问题：

- AI 改对了代码，但改错了需求主线
- AI 写出了功能，但破坏了项目边界
- UI 看起来差不多，但其实没过视觉质量门

## 21. 最后给团队的结论

AI 开发不是“让模型替你写代码”，而是：

1. 你先用 OpenSpec 和文档定义问题
2. 你用 superpowers 把方案和步骤想清楚
3. 你用 task packet 和 AGENTS.md 约束 AI 改动边界
4. 你用 impeccable 和 gstack 保证结果真的过关
5. 你再把成果接回 harness、evidence 和 acceptance

这套方法适合从 0 开始，也适合后面持续做大。

如果只想用一句话讲给团队听，可以直接说：

这不是一套大而全的方法，而是一套低门槛、可理解、可复制的最小协作流程。

### 讲给团队的最后一句话

AI 不是替你做项目管理。
AI 是把“写代码”这一步放大了，但前提仍然是你要有清晰的项目边界、可执行的任务和可验证的结果。
