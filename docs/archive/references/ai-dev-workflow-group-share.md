# 我们现在怎么用 AI 开发项目

> Historical sharing material. This file is for communication context, not current execution routing.
>
> Current workflow routing lives in [Pantheon Workflow Routing](../../methodology/WORKFLOW_ROUTING.md). Older references to `superpowers` or `gstack` as primary flow steps are not active instructions. `impeccable` remains the active UI visual quality gate.

这份文档不是讲 AI 概念，也不是讲某个工具怎么用得更花。

只讲一件事：我们现在怎么把 AI 真正放进项目开发流程里，让它不是“临时问答”，而是能参与设计、开发、验证和验收。

先给结论：

我们现在这套做法，核心不是“让 AI 写代码更快”，而是先把问题、边界、规则和验收口径定清楚，再让 AI 去做事。

大致就是这条线：

`文档/合同 -> 设计/规划 -> task packet -> 代码实现 -> gstack/门禁验证 -> 验收收口`

---

## 先说最容易误解的一点

很多人一提 AI 开发，第一反应是 prompt。

但在真实项目里，prompt 其实只占很小一部分。

真正决定结果的，通常是下面几件事：

- 你有没有把需求边界说清楚
- 项目有没有稳定的文档入口
- AI 知不知道哪些地方能改、哪些地方不能乱改
- 改完之后有没有固定的验证和验收方式

如果这些东西没有，换再强的模型也只是“多一个会写代码的聊天窗口”。

---

## 我们为什么要把文档放在最前面

因为对 AI 来说，文档不是附件，而是正式上下文。

如果文档不清楚，就很容易出现这些问题：

- 同一个需求，不同人问 AI，改动位置完全不同
- AI 把系统域和业务域混着改
- 代码已经写了，但后面发现方向错了
- 做完之后没人说得清到底算不算完成

所以我们现在不是“先写代码，再补文档”，而是先把文档链路定清楚。

在 `pantheon-base` 里，这条主线已经比较明确：

`Contract -> Design -> Assessment -> Remediation -> Acceptance`

这条链的意思很简单：

- `Contract` 定边界、目标、非目标、验收口径
- `Design` 展开具体方案
- `Assessment` 说明实现和设计的差距
- `Remediation` 说明怎么补问题
- `Acceptance` 定义最后怎么验收

这套文档治理不是形式主义，主要是在解决“信息漂移”。

## 我们怎么理解 AI 开发流程

如果用最朴素的话说，AI 开发就是下面这几步：

### 1. 先把问题讲明白

不是直接说“帮我写个页面”，而是先说清楚：

- 这是做什么的
- 给谁用
- 属于哪个模块
- 是底座能力还是业务能力

### 2. 再给 AI 足够的 context

context 不是多给点描述，而是给正确的项目规则。

至少要包含：

- 当前仓库是什么
- 这是 `platform`、`system/*` 还是 `business/*`
- 技术栈和目录结构
- 相关设计文档和合同文档
- 这次改动的验证方式

### 3. 把事情拆小

不要一次让 AI 做完整项目。

比较稳的做法是拆成几段：

- 先设计
- 再接口和数据模型
- 再后端骨架
- 再前端页面
- 最后补测试和验收

### 4. 实现之后必须验证

AI 写完代码不等于完成。

至少要验证：

- 能不能运行
- 接口和页面是否符合预期
- 权限、菜单、i18n 是否接对
- 有没有留下测试、截图、日志或验收记录

### 5. 最后要收口

收口的意思不是“对话结束了”，而是这次工作的结果要能复用、能接手、能回放。

所以最后通常至少要留下：

- 设计依据
- task packet
- 验证命令
- 验收结果

## Harness Engineering 是这套方法的核心

这部分是最值得讲清楚的。

一句话概括：

`coding agent = model + runtime + harness`

意思是：

- `model` 是大模型本身
- `runtime` 是 Claude Code、Codex 这类执行环境
- `harness` 是项目自己定义的一套控制系统

很多团队的问题，不是模型不够强，而是没有 harness。

没有 harness 的时候，AI 开发更像临场发挥：

- 这次这样改
- 下次那样改
- 谁也说不清为什么

有 harness 之后，AI 开发才更像工程交付。

在我们现在这套体系里，harness 主要体现在这些东西上：

- `docs/harness/*`
- `TASK_PACKET_SPEC`
- `VERIFICATION_EVIDENCE_SPEC`
- `REVIEW_LOOP_SPEC`
- `AGENTS.md`

这些东西合在一起，解决的是几个很实际的问题：

- 任务怎么进入系统
- 改动边界怎么定义
- 改完怎么验证
- 结果怎么留证据
- 后续怎么继续

所以 harness 不是“文档负担”，而是为了让 AI 开发可协作、可审计、可回放。

## Superpowers、skills、MCP 分别是什么

这几个词容易混，最简单的理解方式如下。

### 1. Superpowers

superpowers 更像“规划层”。

它不直接负责写代码，主要负责：

- 先定义目标
- 先判断边界
- 先形成设计和计划
- 先把后续约束写清楚

它站在流程前半段：

`需求 -> 设计/计划 -> task packet`

### 2. Skills

skills 更像“方法包”。

它主要解决的是：

“这类事情应该怎么做，按什么套路做更稳”

比如：

- 设计 review
- 代码 review
- QA
- UI 质量检查
- 调试

skill 的价值是把高频动作固化下来，不用每次重新描述。

### 3. MCP

MCP 更像“工具接口层”。

它主要解决的是：

“AI 还能连到哪些外部工具和系统”

比如：

- 浏览器
- 文件系统
- 数据源
- 外部服务

简单总结：

- superpowers 偏规划
- skills 偏方法
- MCP 偏工具连接

## gstack 和前端门禁为什么重要

前端最容易出现“看起来做完了，其实没收口”的情况。

比如：

- 页面能打开，但状态不全
- 文案写死了，没有走 i18n
- 菜单和权限模型没接对
- 页面看着差不多，但没有真实浏览器证据

所以我们现在不会只看“代码写没写出来”，而是会把 gstack 和门禁一起放进流程。

### gstack 主要做什么

gstack 更像执行和验收层，主要做这些事：

- 打开页面
- 注入登录态
- 跑浏览链路
- 看 console error
- 截图
- 采集验收证据

### 前端门禁主要在管什么

本质上是在管这些问题：

- 文案规范
- i18n 约束
- 菜单和路由契约
- 页面状态完整性
- 构建通过
- smoke 通过
- 浏览器证据齐不齐

也就是说，AI 不只是“生成前端代码”，而是要“按前端规则交付前端代码”。

## `AGENTS.md` 为什么这么关键

`AGENTS.md` 不是写给 AI 的一段提示词，它本质上是项目级行为约束。

它的作用很直接：

- 先告诉 AI 应该先读什么
- 再告诉 AI 任务怎么分类
- 再告诉 AI 哪些地方能改，哪些地方不能乱改
- 最后告诉 AI 改完必须补哪些配套项

如果没有这层约束，AI 很容易做出几类典型错误：

- 把 `system/*` 和 `business/*` 混着改
- 把运维业务逻辑塞到底座
- 把认证、IAM、组织、配置当成一个大杂烩模块
- 改了代码，但没同步测试、文档、菜单、权限

在 Pantheon 里，`AGENTS.md` 最重要的作用就是减少这种“错误自由度”。

它要求 AI 先做两件事：

### 1. 先判断归属层

也就是先判断这是：

- `platform`
- `system/auth`
- `system/iam`
- `system/org`
- `system/config`
- `business/*`

很多问题不是不会写，而是改错层了。

### 2. 先理解 base 和 ops 的边界

在 `pantheon-base` 里：

- 平台层负责应用壳、聚合页、路由装配
- 系统域负责认证、IAM、组织、配置、审计等通用能力
- 业务域通过 `business/*` 扩展

在 `pantheon-ops` 里：

- `pantheon-base` 是唯一底座知识源
- 通用能力优先回 base 处理
- ops 只放自己的运维业务模块

这个边界一旦不清楚，AI 就很容易越改越乱。

## Claude Code 和 Codex 放在这套体系里怎么理解

对我们来说，Claude Code 和 Codex 都不是“单独一套方法”。

它们更像两种执行环境，接到同一套仓库协议上。

也就是说，重点不在“你选哪个工具”，而在：

- 有没有稳定的文档入口
- 有没有明确的边界约束
- 有没有 task packet
- 有没有验证和验收门禁

工具可以换，但流程不能乱。

## 放到我们当前项目里，最简单的理解方式

### `pantheon-base` 在做什么

`pantheon-base` 是底座，不是具体业务应用。

它负责的是：

- 平台层
- 系统域
- 通用后台能力
- 低代码治理和模块接入治理

一句话，就是“后台系统通用底座”。

### `pantheon-ops` 在做什么

`pantheon-ops` 是在底座上做运维业务。

当前最重要的两块业务是：

- CMDB
- Deploy

也就是说：

- base 负责地基
- ops 负责运维业务
- 通用能力回到底座
- 业务差异留在 ops

## 一个贴近我们项目的例子

假设我们现在要在 `pantheon-ops` 里新增一个“主机列表”页面。

这件事如果按我们现在的方法来做，大致会是这样：

### 第一步：先判断边界

先回答几个问题：

- 这是通用底座能力，还是运维业务能力
- 数据表该不该放在 `biz_*`
- 权限、菜单是不是复用底座现有模型

这个例子里，答案应该是：

- 属于 `business/cmdb`
- 数据表应该走 `biz_*`
- 权限和菜单复用底座现有能力

### 第二步：先找设计依据

不是直接进代码，而是先看：

- 相关 contract
- 相关 design
- ops 自己的业务设计
- `AGENTS.md`

### 第三步：再交给 AI 做具体任务

这时才适合写明确任务：

- 先出数据模型和接口设计
- 再出后端模块骨架
- 再出前端页面骨架
- 再补菜单、权限、i18n
- 最后补测试和验收

### 第四步：最后人工看什么

重点不是“代码有没有生成”，而是：

- 有没有挂在正确模块下
- 有没有破坏 base 和 ops 边界
- 权限和菜单是不是走现有机制
- 命名和结构是不是符合项目约束
- 有没有留下验证和验收证据

这个例子其实就是我们现在最常见的 AI 协作方式。

## 我们现在这套方法，最想解决什么问题

最核心就是两件事：

### 1. 不让 AI 乱改

通过：

- 文档治理
- `AGENTS.md`
- harness
- task packet

把边界和流程钉住。

### 2. 不让“写出来了”冒充“交付完成”

通过：

- gstack
- 前端门禁
- smoke
- evidence
- acceptance

把“代码完成”变成“结果可验证”。

---

## 最后一句话

我们现在不是在追求“AI 自动写项目”。

我们是在做一套更稳的开发方式：

- 先把文档和边界定清楚
- 再把计划和任务拆清楚
- 再让 AI 去做
- 最后用门禁和验收把结果收口

如果只看 prompt，这套东西会显得很重。

但只要进入真实项目，就会发现真正有价值的，恰恰不是 prompt，而是这整套从设计到验收的工程闭环。
